# System Contract

시스템 경계, 책임, 불변 조건(변경 금지 규칙)을 정의합니다.
이 문서는 구현 변경 시 "무엇을 지켜야 하는지"를 빠르게 판단하는 기준입니다.

## 목적

- 모듈 간 책임 경계를 명확히 유지
- 변경 시 회귀(regression) 위험 최소화
- 신규 참여자 온보딩 시 시스템 제약 빠른 전달

## 계약 범위

- 런타임/애플리케이션 계층 경계
- 공개 인터페이스(API, 이벤트, CLI) 안정성
- 데이터 정합성/에러 처리/관측 가능성(로깅, 모니터링) 원칙

## 아키텍처 불변 조건 (Invariants)

아래 항목은 합의 없이 깨뜨리지 않습니다.

1. 공개 API 시그니처는 호환성 검토 없이 변경하지 않는다.
2. 비즈니스 규칙은 서비스 계층에 두고, 입출력 계층에 분산하지 않는다.
3. 외부 I/O(네트워크/파일/DB)는 경계 계층을 통해서만 접근한다.
4. 실패 가능한 경로는 명시적 에러 처리와 로그를 남긴다.
5. 테스트 가능한 단위 경계를 유지한다(핵심 로직 단위 테스트 가능해야 함).

## 계층 책임

| 계층 | 책임 | 금지 사항 |
| ---- | ---- | ---- |
| Interface (API/CLI) | 입력 검증, 응답 포맷 | 비즈니스 규칙 직접 구현 |
| Application/Service | 유스케이스 오케스트레이션, 정책 적용 | 인프라 상세 직접 의존 |
| Domain (선택) | 핵심 규칙/모델 | 외부 I/O 접근 |
| Infrastructure | DB/외부 API/파일 시스템 연동 | 정책 결정 로직 포함 |

## 호환성 정책

- 하위 호환성 깨지는 변경은 사전 합의 후 진행
- 공개 인터페이스 변경 시 `.workflowSvc/docs/api/api-spec.md`와 `.workflowSvc/docs/changelog.md` 동시 업데이트
- 마이그레이션 필요 시 단계적 전환 경로를 문서화

## 변경 절차

1. 변경 이유와 영향 범위를 `.workflowSvc/workflow/decisions.md`에 기록
2. 필요 시 ADR(`.workflowSvc/docs/decisions/`)로 결정 승격
3. 구현 후 `.workflowSvc/workflow/3.review.md` 체크리스트 검증
4. `docs:finalize` 및 관련 문서 반영

## 예외 처리

- 긴급 장애 대응으로 계약 위반이 필요한 경우, 임시 조치 후 복구 계획을 함께 문서화
- 임시 우회 로직에는 제거 조건(기한/완료 조건)을 반드시 남김
