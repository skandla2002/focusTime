### ISSUE-032: AdMob 실제 ID 교체 — 출시 전 테스트 ID를 실제 ID로 교체

> 출처: ISSUE-031 해결 중 AdMob App ID 구조 파악 — 테스트 ID로 임시 운영 중, 실제 출시 전 교체 필요

- 배경:
  - 현재 코드에는 Google 공식 테스트 ID가 3곳에 적용되어 있음
  - 실제 출시 시 AdMob 콘솔에서 발급받은 ID로 교체해야 실제 광고 수익 발생
  - AdMob ID 체계:

    | ID 종류 | 형식 | 용도 |
    | ------- | ---- | ---- |
    | **App ID** | `ca-app-pub-XXXX~XXXX` | AdMob SDK 초기화용 (앱당 1개) |
    | **배너 Ad Unit ID** | `ca-app-pub-XXXX/XXXX` | 배너 광고 슬롯 (광고 종류별 1개) |
    | **전면 Ad Unit ID** | `ca-app-pub-XXXX/XXXX` | 전면 광고 슬롯 (광고 종류별 1개) |

- 구현 내용:

  **Step 1: AdMob 콘솔에서 ID 발급**

  1. [admob.google.com](https://admob.google.com) → Google 계정 로그인
  2. **앱 추가** → 앱 이름 입력 → Android 선택 → **앱 ID 확인** (`ca-app-pub-XXXX~XXXX`)
  3. **광고 단위 만들기** → 배너 광고 생성 → **배너 Ad Unit ID 확인**
  4. **광고 단위 만들기** → 전면 광고 생성 → **전면 Ad Unit ID 확인**

  **Step 2: .env 파일 생성 (프로젝트 루트)**

  ```env
  VITE_ADMOB_BANNER_ID_ANDROID=ca-app-pub-발급받은값/배너AdUnitID
  VITE_ADMOB_INTERSTITIAL_ID_ANDROID=ca-app-pub-발급받은값/전면AdUnitID
  ```

  > ⚠ `.env` 파일은 `.gitignore`에 추가 — 절대 커밋하지 않음

  **Step 3: AndroidManifest.xml App ID 교체**

  파일: `android/app/src/main/AndroidManifest.xml`

  ```xml
  <meta-data
      android:name="com.google.android.gms.ads.APPLICATION_ID"
      android:value="ca-app-pub-발급받은앱ID~숫자" />
  ```

  **Step 4: capacitor.config.ts App ID 교체**

  파일: `capacitor.config.ts`

  ```ts
  android: 'ca-app-pub-발급받은앱ID~숫자',
  ```

  **Step 5: 재빌드 및 테스트**

  ```powershell
  npx cap sync android
  npx cap build android --keystorepath upload-keystore.jks --keystorepass 'eptest$00' --keystorealias upload --keystorealiaspass 'eptest$00'
  ```

- 수정 대상 파일:
  - [ ] `.env` (신규 생성 — gitignore 추가 필수)
  - [ ] `android/app/src/main/AndroidManifest.xml` — App ID 실제 값으로 교체
  - [ ] `capacitor.config.ts` — android App ID 실제 값으로 교체
- ⚠ 참고/제약사항:
  - `BannerAd.tsx`, `InterstitialAd.tsx`의 Ad Unit ID는 `.env` 변수를 우선 읽으므로 코드 수정 불필요
  - `.env` 없으면 코드 내 테스트 ID가 자동 fallback으로 사용됨
  - AdMob 계정 승인까지 수일 소요될 수 있음 — 승인 전에는 테스트 ID 유지
- 테스트 필요: Yes — 실제 광고 표시 확인
- 완료 기준: 앱 실행 시 실제 AdMob 광고 표시 확인

### ISSUE-055: 집중 잠금 시 포커스 실드 레이어 추가 — 비필수 UI 어둡게, 타이머 중심 시선 유도

> 출처: brief.md — 집중모드에 '잠금'을 하면 전체 화면을 어둡게 감싸 시선 방해를 줄이고 싶음. 더 좋은 방법이 있다면 함께 정리 요청

- ⏱ 예상: AI 60분 / 시니어 30분 | 🤖 Subagent: 없음

- 배경 및 방향:
  - 현재 `focusLock`은 버튼 비활성화 중심이라 시각적으로는 일반 상태와 큰 차이가 없음
  - 화면 전체를 완전히 가리는 검은 레이어는 타이머 확인과 제어까지 방해해 오히려 사용성이 떨어질 수 있음
  - 권장 방향은 "전체 차단 모달"보다 `포커스 실드(focus shield)` 방식:
    - 비필수 영역은 어둡게/부드럽게 가라앉힘
    - 타이머 링, 남은 시간, 시작·일시정지·리셋·잠금 버튼은 위로 띄워 계속 읽고 누를 수 있게 유지
    - `pointer-events: none` 기반 시각 레이어로 처리해 기존 잠금 제어 로직과 충돌하지 않게 구성

- 구현 내용:

  **`src/screens/TimerScreen.tsx` 수정**

  - `focusLock && mode === 'focus'`일 때만 포커스 실드 레이어 렌더링
  - 타이머 핵심 영역에 `focusPriority` 클래스를 부여해 overlay 위에 배치
  - 필요하면 짧은 상태 문구(예: "집중 잠금 중")를 추가하되, 시선을 빼앗지 않는 수준으로 최소화

  **`src/screens/TimerScreen.module.css` 수정**

  - 화면 전체를 덮는 반투명 overlay + radial gradient / vignette 스타일 추가
  - 핵심 UI는 높은 z-index와 약한 glow 또는 대비 강화로 강조
  - `prefers-reduced-motion` 환경에서는 불필요한 애니메이션 제거

- 수정 대상 파일:
  - [ ] `src/screens/TimerScreen.tsx` — 포커스 실드 레이어 및 우선순위 DOM 연결
  - [ ] `src/screens/TimerScreen.module.css` — overlay / spotlight / 강조 스타일
  - [ ] `src/screens/TimerScreen.test.tsx` — 잠금 상태에서 overlay 노출 및 핵심 버튼 접근 가능 여부 테스트

- 참고/제약사항:
  - `completionOverlay`, 메모 모달, 인터스티셜보다 위아래 레이어 우선순위가 꼬이지 않도록 z-index 체계 점검 필요
  - 배너 광고나 GNB까지 추가로 가리기보다, TimerScreen 내부의 비핵심 영역 디밍에 우선 집중
  - "더 좋은 방법"에 대한 판단은 전체 암전보다 핵심 UI만 남기는 spotlight 방식이 UX상 더 적합한 것으로 계획

- 테스트 필요: Yes — 잠금 ON + focus 모드에서 overlay 표시, 시작/계속 버튼과 잠금 버튼 접근 가능 여부 확인
- 완료 기준:
  - focusLock ON + focus 모드에서 비핵심 영역이 눈에 띄게 어두워짐
  - 타이머 시간과 핵심 제어 버튼은 시인성과 조작 가능 상태 유지
  - break 모드 또는 잠금 OFF 상태에서는 포커스 실드가 나타나지 않음

### ISSUE-056: 시각 모드 토큰 인프라 추가 — color / grayscale 테마 토큰 + 상태 저장

> 출처: brief.md — 흑백 모드를 추가하되 디자인 token부터 우선 추가하고, 이후 스타일 개선까지 고려해 달라는 요청

- ⏱ 예상: AI 90분 / 시니어 45분 | 🤖 Subagent: 없음

- 배경:
  - 현재 토큰 파이프라인은 `:root` 기준 단일 컬러 세트만 생성함
  - 단순 `filter: grayscale(1)`은 차트, 강조색, 광고 placeholder, 텍스트 대비를 일괄 훼손할 수 있어 유지보수와 접근성 면에서 불리함
  - 요청대로 "token first"로 가려면 색상 의미를 semantic token으로 재구성하고, 흑백 모드 전용 변형 값을 토큰 레벨에서 제공해야 함

- 구현 내용:

  **토큰 구조 개편**

  - `src/tokens/design-tokens.json`에 시각 모드별 semantic token 추가
    - 예: `surface`, `surfaceElevated`, `textPrimary`, `textSecondary`, `accent`, `accentMuted`, `border`, `chartPrimary`
  - color / grayscale 두 세트를 같은 의미 체계로 관리

  **토큰 생성기 확장**

  - `scripts/generate-tokens.mjs`를 확장해
    - 기본 `:root`는 color 모드 변수 출력
    - `[data-visual-mode='grayscale']` 선택자에는 grayscale 변수 override 출력

  **앱 상태 추가**

  - `src/store/appStore.ts`에 `visualMode: 'color' | 'grayscale'` 및 setter 추가
  - 사용자 선택을 localStorage에 저장해 새로고침 후에도 유지
  - `src/App.tsx` 루트 컨테이너에 `data-visual-mode` 속성 연결

- 수정 대상 파일:
  - [ ] `src/tokens/design-tokens.json` — 시각 모드별 semantic token 정의
  - [ ] `scripts/generate-tokens.mjs` — grayscale override CSS 생성 로직 추가
  - [ ] `src/styles/_tokens.css` — 토큰 재생성 결과 반영
  - [ ] `src/store/appStore.ts` — visualMode 상태 및 persistence
  - [ ] `src/store/appStore.test.ts` — visualMode 변경/저장 테스트
  - [ ] `src/App.tsx` — 루트 data attribute 연결

- 참고/제약사항:
  - 기존 CSS가 일부 raw color 또는 구형 변수명에 의존하고 있으면 semantic token 쪽으로 순차 치환 필요
  - grayscale 팔레트는 완전 무채색만 고집하기보다 대비와 계층이 유지되는 "저채도 neutral" 범위까지 허용하는 것이 실사용에 유리
  - 차트/강조색도 토큰화 대상에 포함해야 후속 ISSUE-057에서 일관 적용 가능

- 테스트 필요: Yes — appStore visualMode 변경 및 초기값 복원 테스트
- 완료 기준:
  - 앱 루트에 `data-visual-mode="color|grayscale"`가 반영됨
  - 흑백 모드용 토큰이 생성 파이프라인을 통해 CSS 변수로 출력됨
  - 사용자가 시각 모드를 바꾸면 새로고침 후에도 유지됨
